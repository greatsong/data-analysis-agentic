# 데이터 분석 에이전틱 수업 시리즈: 총괄 문서

## "AI가 분석을 대신하는 시대에, 학생은 무엇을 배워야 하는가"

---

## 1. 이 수업 시리즈의 출발점

### 1.1 문제 인식

2025년 12월, AI 인프라 회사 Lablup의 대표가 40일 만에 AI와 협업하여 100만 줄짜리 소프트웨어를 완성했다. 같은 규모를 예전에는 3년 걸려 만들었다. 코드를 직접 짜는 능력의 경제적 가치가 급격히 변하고 있다.

이 변화는 데이터 분석 영역에도 그대로 적용된다. AI에게 "이 데이터 분석해줘"라고 말하면, 10초 안에 상관분석, 시각화, 해석 요약이 나온다. 학생이 판다스 코드를 외우고 matplotlib 문법을 익히는 데 수업 시간 대부분을 쓰는 구조는, AI 시대에 "타이핑 연습"에 해당할 위험이 있다.

그렇다면 데이터 수업에서 무엇을 가르쳐야 하는가?

### 1.2 핵심 답변

```
AI는 데이터 분석을 "실행"하지만,
분석을 "설계"하고 "검증"하는 것은 사람의 몫이다.

그리고 설계의 출발점은 코드가 아니라 질문이다.

현실의 모호한 상황에서
데이터로 답할 수 있는 질문을 만들어내는 능력 —
이것이 AI 시대에도 변하지 않는 근본 역량이며,
이 수업 시리즈가 키우고자 하는 것이다.
```

## 2. 교육 철학과 설계 원리

### 2.1 세 가지 철학적 전제

**전제 1: 도구가 바뀌어도 사고는 남는다**

판다스든 AI든, 도구는 바뀐다. 하지만 "모호한 상황에서 분석 가능한 질문을 구성하는 능력", "결과를 비판적으로 평가하는 능력", "데이터의 한계를 인식하는 능력"은 도구와 무관하다. 이 수업은 도구 사용법이 아니라 사고 방식을 가르친다.

**전제 2: 코드를 안 가르치는 것이 아니라, 코드의 위치를 재배치하는 것이다**

코드를 이해하는 능력은 여전히 필요하다. AI가 만든 분석 결과를 검증하려면 코드를 읽을 줄 알아야 한다. 다만 "코드를 처음부터 끝까지 직접 타이핑하는 것"에 수업 시간 대부분을 쓰는 것을 재고한다. 비유하자면 — 계산기가 나왔다고 수학 교육이 사라지지 않았듯이, AI가 코드를 짠다고 컴퓨팅 사고력 교육이 사라지지 않는다. 사고력이 발현되는 방식이 달라질 뿐이다.

**전제 3: 결과물이 아니라 생성기를 만들어라**

에이전틱 워크플로우의 핵심 원칙은 "결과물을 직접 수정하지 말고, 결과물을 만드는 지침(생성기)을 수정하라"이다. 이것을 교육에 적용하면, 학생이 평가받는 것은 "분석 결과의 정확성"이 아니라 "분석을 설계하는 지침서의 질"과 "결과를 비판적으로 검증하는 과정"이다.

### 2.2 네 가지 설계 원리

**원리 1: 질문이 먼저다 (Question-First Design)**

기존 데이터 수업은 데이터가 먼저 주어지고 학생은 "처리"한다. 이 수업은 질문이 먼저이고 데이터는 역으로 설계한다. "이 데이터를 분석하시오"가 아니라 "이 상황에서 무엇을 데이터로 확인할 수 있는가?"를 묻는다. 이 순서의 전환이 수업 전체의 뼈대이다.

**원리 2: 스캐폴딩의 점진적 제거 (Gradual Release)**

1차시에서 교사가 상황카드와 구조를 주고, 2차시에서 데이터를 주고, 3차시에서 검토 프레임을 준다. 4차시에서는 아무것도 주지 않는다. 학생이 독립적으로 전체 사이클을 돌릴 수 있는가 — 이것이 최종 도달점이다.

```
1차시: 교사가 상황을 준다 → [질문 설계] ← 학생이 한다
2차시: 교사가 데이터를 준다 → [AI 실행+검증] ← 학생이 한다
3차시: 교사가 프레임을 준다 → [비판적 검토] ← 학생이 한다
4차시: 교사가 아무것도 안 준다 → [전체 사이클] ← 학생이 혼자 한다
```

**원리 3: 검증이 실행보다 중요하다 (Verification Over Execution)**

AI가 분석 결과를 완벽하게 줬어도 검증을 형식적으로 한 학생보다, AI 결과에서 오류를 발견하고 지침을 개선한 학생이 높은 평가를 받는 구조를 설계했다. 이 평가 구조가 학생에게 "분석의 실행이 아니라 설계와 검증이 핵심"이라는 메시지를 일관되게 전달한다.

**원리 4: 한계 명시는 역량이다 (Limitations as Competence)**

"이 분석의 한계는 없습니다"는 가장 나쁜 답이다. 데이터의 한계, 수집 방법의 편향, 해석의 범위를 자발적으로 명시할 수 있는 것이 데이터 리터러시의 정점이다. 매 차시에서 한계 인식을 반복적으로 요구한다.

## 3. 성취기준 연결

### 3.1 2022 개정 교육과정 '정보' 데이터 영역과의 대응

| 교육과정 핵심 아이디어 | 이 수업에서의 구현 |
|---|---|
| 디지털 세상의 데이터는 다양한 형태로 표현된다 | 1차시: 모호한 현실 상황에서 "어떤 형태의 데이터가 필요한가"를 역설계 |
| 목적에 맞게 수집·분석·시각화한다 | 2차시: AI에게 분석을 지시하는 구조화된 지침서 설계 + 시각화 적절성 검증 |
| 유용한 정보를 도출한다 | 3차시: 동료 피어리뷰를 통한 해석의 타당성 검증 / 4차시: 분석 결과의 의사결정 연결 |

### 3.2 교과 역량 연결

| 교과 역량 | 이 수업에서의 발현 |
|---|---|
| 컴퓨팅 사고력 | 추상화(모호함→구체적 질문), 분해(큰 문제→작은 질문), 자동화(AI 지시 설계) |
| 디지털 문화 소양 | AI 결과의 비판적 수용, 데이터 한계 인식, AI와의 건강한 협업 |
| 인공지능 소양 | AI를 분석 도구로 활용하되, 설계와 검증은 인간이 수행하는 협업 구조 이해 |
| 협력적 소통 | 3차시 피어리뷰: 근거 기반 비판, 피드백 수용/반론, 조 간 교차 검토 |

### 3.3 교사용 지도서 질문의 전환

```
기존:
  "학생이 이 데이터를 수집할 수 있는가?"
  "학생이 이 코드를 작성할 수 있는가?"
  "학생이 이 그래프를 그릴 수 있는가?"

이 수업:
  "학생이 이 상황에서 분석 가능한 질문을 만들 수 있는가?"
  "학생이 AI의 분석 결과를 비판적으로 검증할 수 있는가?"
  "학생이 자기 분석의 한계를 스스로 명시할 수 있는가?"
  "학생이 전체 사이클을 독립적으로 수행할 수 있는가?"
```

### 4.2 차시별 핵심 설계 의도

| 차시 | 핵심 활동 | 컴퓨터 | 교사의 역할 | 학생이 넘는 허들 |
|---|---|---|---|---|
| 1차시 | 질문 3단 변환 + 데이터 역설계 | 없음 | 상황카드 제공, 변환 시연, 순회 촉진 | "느낌"을 "측정 가능한 질문"으로 바꾸기 |
| 2차시 | 지침서 작성 → AI 실행 → 검증 → 수정 | 사용 | 나쁜 지시 vs 좋은 지시 시연, 순회 지도 | AI 결과를 그대로 받지 않고 검증하기 |
| 3차시 | 동료 피어리뷰 + 수용/반론 + 재분석 | 사용 | 교환 규칙 안내, 토론 촉진 | 남의 분석에서 약점 찾기, 피드백 수용하기 |
| 4차시 | 자유 주제 전체 사이클 독립 수행 | 사용 | 최소 개입 (주제 고민 시에만 개입) | 혼자서 전체 사이클 돌리기 |

### 4.3 차시 간 연결 구조

각 차시의 산출물이 다음 차시의 입력이 됩니다.

```
1차시 산출물                     2차시 입력
  Part A: 분석 질문      →       지침서(Soul Document)의 ①②
  Part B: 데이터 설계     →       지침서의 ③④⑤

2차시 산출물                     3차시 입력
  지침서 + AI 결과 + 검증 기록 →  상대 조 검토 대상

3차시 산출물                     4차시 준비
  피어리뷰 경험 + "좋은 분석" 기준 →  독립 실행의 내면화된 기준
```

1차시를 빠뜨리면 2차시에서 "분석해줘" 수준의 지시밖에 못합니다. 2차시 없이 3차시를 하면 검토할 대상이 없습니다. 3차시 없이 4차시를 하면 자기 검증 기준이 약합니다. 네 차시는 독립이 아니라 누적입니다.

## 5. 평가 체계

### 5.1 평가 철학

```
이 수업 시리즈의 평가는 "분석 결과의 정확성"이 아니라
"분석 설계의 구조화 정도"와 "비판적 사고의 깊이"를 평가합니다.

AI가 완벽한 결과를 줬어도 검증을 형식적으로 한 학생 <
AI 결과에서 오류를 발견하고 지침을 개선한 학생

분석 질문이 데이터로 답할 수 없는 학생 <
모호한 질문에서 출발하여 측정 가능한 질문으로 변환한 학생

한계를 "없다"고 쓴 학생 <
한계를 구체적으로 명시하고도 유의미한 결론을 내린 학생
```

### 5.2 차시별 평가 요소 및 배점

| 반복 개선(지침 수정→재실행→개선 확인) | | ★ (20점) | ★ (20점) | |

### 5.3 시리즈 종합 평가 구조 (수행평가 활용 시)

4차시 전체를 하나의 수행평가 단위로 운영할 경우:

| 영역 | 비중 | 근거 |
|---|---|---|
| 질문 설계 역량 | 30% | 1차시 Part A + 4차시 Part A |
| 데이터 설계 역량 | 15% | 1차시 Part B |
| AI 협업 역량 (지시 설계 + 검증 + 개선) | 25% | 2차시 전체 + 4차시 Part C |
| 비판적 사고 (피어리뷰 + 한계 인식) | 20% | 3차시 전체 + 4차시 Part D |
| 성찰 및 메타인지 | 10% | 전 차시 성찰 종합 |

### 5.4 평가 시 유의사항

- **"결과의 정확성"은 평가하지 않습니다.** AI가 만든 결과의 정확성은 AI의 성능에 의존하며, 학생의 역량이 아닙니다. 학생이 평가받는 것은 "어떤 질문을 했는가", "결과를 어떻게 검증했는가", "한계를 인식했는가"입니다.
- **"코드 작성 능력"은 직접 평가하지 않습니다.** 다만 2차시 검증 과정에서 AI가 만든 코드를 읽고 논리적 오류를 발견하는 능력은 간접적으로 평가됩니다.
- **같은 상황카드에서 다른 질문이 나오는 것은 정상입니다.** 질문의 "유일한 정답"은 없습니다. 좋은 질문과 덜 좋은 질문의 차이는 있으며, 그 기준이 루브릭에 명시되어 있습니다.
- **4차시 미완성은 감점이 아닙니다.** 32분 안에 전체 사이클을 완벽하게 도는 것은 어렵습니다. "어디까지 갔는가"와 "가지 못한 이유를 인식하는가"를 함께 평가합니다.

## 6. 교사 운영 가이드

### 6.1 사전 준비 체크리스트

| 준비물 | 1차시 | 2차시 | 3차시 | 4차시 |
|---|---|---|---|---|
| 상황카드 6종 인쇄 | ✓ | | | |
| 학생 활동지 인쇄 | ✓ | ✓ | ✓ | ✓ |
| PC/노트북 + AI 도구 접근 | | ✓ | ✓ | ✓ |
| CSV 데이터 파일 | | ✓ | | |
| 공개 데이터 목록 링크 | | | | ✓ |
| 나쁜 지시 vs 좋은 지시 시연 준비 | | ✓ | | |

### 6.2 교사 개입 수준 가이드

```
1차시: 적극 개입
  - 3단 변환 시연을 상세히 보여줌
  - 순회하며 "네가 궁금한 게 뭔데?"로 질문 구체화 촉진
  - 데이터 역설계에서 [5]번 함정을 반드시 쓰도록 안내

2차시: 중간 개입
  - 도입에서 나쁜 지시 vs 좋은 지시 시연
  - 순회하며 검증이 형식적인 학생에게 "정말? 한 가지라도 의심스러운 점은?"
  - "결과물 말고 지침서를 고쳐" 원칙을 반복 강조

3차시: 촉진 위주
  - 교환 규칙과 검토 프레임 안내
  - 피드백 전달 시 "구체적으로"를 강조
  - 수용/반론 시 "근거"를 요구

4차시: 최소 개입
  - 주제 고민 10초 이상 → "지금 가장 궁금한 것?"
  - 데이터 5분 이상 못 찾으면 → 목록 가리키며 "여기서 하나 골라"
  - AI 결과에 만족할 때 → "한계는?" (한 마디만)
  - 그 외에는 개입하지 않음
```

### 6.3 시간 조정 가이드

50분 기준으로 설계되었으나, 학교 실정에 따라 조정이 필요할 수 있습니다.

**45분 블록 시**: 각 차시에서 성찰(Part D/E) 시간을 축소하고 다음 차시 도입에 통합합니다.

**100분 블록 시**: 1+2차시, 3+4차시를 묶어 2블록으로 운영하면 차시 간 연결이 자연스럽습니다. 이 경우 1→2 사이의 쉬는 시간에 "1차시 질문이 2차시 지침서의 입력이 된다"는 연결을 명시적으로 안내합니다.

**3차시만 독립 운영이 어려울 때**: 2차시 후반에 간이 교차 검토(10분)를 삽입하고, 3차시 내용을 4차시 도입에 축약 통합할 수 있습니다.

### 6.4 상황카드 커스터마이징

제공된 6종의 상황카드는 일반적인 고등학교 맥락입니다. 학교 실정에 맞게 교체하는 것을 강력히 권장합니다. 효과적인 상황카드의 조건:

- **모호해야 합니다.** "급식이 맛없는 것 같다"처럼 느낌 수준이어야 합니다. "급식 만족도가 3.2점이다"는 이미 데이터가 있으므로 질문 설계 연습이 안 됩니다.
- **학생 생활과 연결되어야 합니다.** 추상적 사회 문제보다 학생이 체감하는 학교 내 상황이 더 몰입도가 높습니다.
- **의사결정으로 연결 가능해야 합니다.** 분석 결과에 따라 실제로 무언가를 바꿀 수 있는 상황이어야 3단계 변환이 의미를 가집니다.

## 7. 이론적 배경

### 7.1 에이전틱 워크플로우와 교육

이 수업 시리즈는 AI Frontier EP 86 "진짜 내 일을 위한 Agentic Workflow"(2026.02.15)에서 논의된 에이전틱 워크플로우의 교육적 함의를 구체적 수업으로 번역한 것입니다.

에이전틱 워크플로우에서 인간의 역할은 네 가지로 정리됩니다:

1. **문제 정의** — 무엇을 해결할 것인가 (이 수업의 1차시)
2. **구조 설계** — 어떤 순서로, 어떤 지침으로 AI에게 맡길 것인가 (2차시)
3. **품질 검증** — AI 결과가 맞는가, 충분한가 (2-3차시)
4. **자기 관리** — 지속 가능하게 AI와 협업하는 방법 (전 차시 성찰)

이 네 가지는 순서대로 1~4차시에 대응하되, 4차시에서 전체가 통합됩니다.

### 7.2 "질문 설계"의 학문적 근거

데이터 과학에서 질문 설계(Problem Framing)는 분석의 첫 번째이자 가장 중요한 단계로 인정받고 있습니다. 통계학에서는 이를 "연구 질문의 조작화(Operationalization)"라 부르며, 모호한 개념을 측정 가능한 변수로 변환하는 과정입니다.

이 수업의 "3단 변환"은 이 과정을 고등학생 수준으로 구조화한 것입니다:

```
학문적 과정:              이 수업의 3단 변환:
  
  개념 정의              →  1단계: 모호한 느낌 → 구체적 현상
  조작화(측정 가능한 변수)  →  2단계: 현상 → 데이터로 답할 수 있는 질문
  가설→검정→해석           →  3단계: 질문 → 의사결정 연결
```

### 7.3 비판적 사고와 피어리뷰

3차시의 동료 피어리뷰는 과학적 방법론의 핵심인 "동료 검토(Peer Review)"를 교실에 이식한 것입니다. 연구에 따르면, 타인의 작업을 평가하는 경험이 자기 작업의 질을 높이는 데 직접적으로 기여합니다. 특히 "근거 기반 비판"을 요구하는 것이, 단순한 "좋다/나쁘다" 평가보다 학습 효과가 높습니다.

## 9. 관통 질문

이 수업 시리즈 전체를 관통하는 질문은 4차시 마지막 성찰에 있습니다:

```
"AI가 있는 시대에,
 데이터 분석에서 사람이 반드시 해야 하는 것은 무엇인가?"
```

이 질문에 대한 답을 교사가 알려주지 않습니다. 4차시에 걸친 경험을 통해 학생이 스스로 도달합니다. 우리가 기대하는 답은 이런 방향입니다:

```
AI는 분석을 실행하지만,
어떤 질문을 던질 것인가를 정하는 것,
결과가 맞는지 따지는 것,
이 분석의 한계가 무엇인지 아는 것,
그리고 이 결과로 무엇을 할지 결정하는 것 —
이것은 사람의 몫이다.
```

하지만 이 문장을 외우게 하는 것이 아니라, 학생이 자기 경험에서 자기 언어로 이것을 발견하는 것이 이 수업 시리즈의 최종 목표입니다.
